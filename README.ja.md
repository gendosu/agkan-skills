# agkan-skills

agkan CLI ツールを使用した、Claude Code での タスク管理ワークフロー用の包括的なスキルコレクション。

## 概要

agkan-skills プラグインは、agkan という SQLite ベースのタスク管理ツールを使用して開発ワークフローを管理するための8つの強力なスキルを提供します。agkan は AI エージェントのコラボレーションに最適化されており、構造化されたカンバンワークフローを通じて効率的なタスク計画、実行、進捗追跡を実現します。

## 機能

- **タスク管理**: タスクの作成、一覧表示、更新、階層的な関係での整理
- **計画ワークフロー**: バックログタスクの確認と評価、分解と準備状態の確認
- **タスク実行**: 優先度の高いタスクを選択し、実装してプルリクエストを作成
- **ブロッキング関係**: タスク依存関係とブロッキング関係の定義
- **メタデータサポート**: タスクに優先度レベルとカスタムメタデータを付与
- **JSON出力**: プログラム処理用の機械可読出力

## スキル

### 1. agkan

agkan CLI ツールを使用したタスク管理のコアスキル。

**用途:**
- タスクの作成、一覧表示、更新
- タスク階層管理（親タスク・子タスク関係）
- タスクステータスの設定（backlog → ready → in_progress → review → done → closed）
- タグとブロッキング関係の管理
- カンバンボードによるプロジェクト進捗の追跡

**主要コマンド:**
```bash
# タスク操作
agkan task add "タイトル" "本文"
agkan task list --status ready
agkan task update <id> status in_progress
agkan task get <id> --json

# タグとメタデータ
agkan task tag add "feature"
agkan task meta set <id> priority high
```

### 2. agkan-planning

開発開始前に、バックログタスクを確認し、分解度、実装準備状態、優先順位の順序を評価します。

**用途:**
- バックログタスクの内容確認と完全性の確認
- タスクのさらなる細分化が必要かどうかの判断
- 準備完了したタスクを「ready」ステータスに移行
- 保留すべきタスクに「いつかやる」タグを付与
- タスク要件を理解するためのコード調査

**典型的なフロー:**
1. すべてのバックログタスクを取得
2. 各タスクについて:
   - 内容が明確か確認する（必要に応じてコードを調査）
   - 分解が必要かどうか評価する（1タスク = 1PR = 1機能）
   - 実装が明確で障害物がない場合、「ready」ステータスに移行
   - 保留可能な場合、「いつかやる」タグを付与

### 3. agkan-run

「ready」キューから最も優先度の高いタスクを選択し、実装してプルリクエストを作成し、完了とします。

**用途:**
- ready キューから最も優先度の高い次のタスクを選択
- 単一の焦点を絞ったタスクで開発セッションを開始
- 構造化された実行ワークフローに従う

**優先度の基準:**
1. 重要度レベル（高 → 中 → 低）
2. タグ優先度（bug → security → improvement → test → performance → refactor → docs）
3. 子タスクとブロッカーが優先化される

**典型的なフロー:**
1. すべての ready タスクを取得
2. 重要度とタグ階層を使用して優先度を評価
3. 最も優先度の高いタスクを選択
4. サブエージェントに実装を委譲
5. タスクを完了とするか、次のタスクに移行

### 4. agkan-subtask

選択されたタスクを単独で実装し、in_progress ステータスの更新、ブランチ作成、実装、プルリクエスト作成、完了処理を行います。

**用途:**
- 特定の選択されたタスクを実装
- 完全な開発ライフサイクル管理（ブランチ作成から PR まで）
- コード変更、コミット、プルリクエスト作成の処理
- 開発ワークフロー全体を通じてタスクステータスを更新

**重要:**
- このスキルはユーザーが直接呼び出すことはできません
- agkan-run スキルにより内部的に使用されます
- タスクが事前に選択されていることが前提です

**典型的なフロー:**
1. タスクステータスを「in_progress」に更新
2. フィーチャーブランチを作成
3. タスクを実装
4. プルリクエストを作成して提出
5. タスクステータスを「review」に更新

### 5. agkan-run-direct

「ready」キューから最も優先度の高いタスクを選択し、ブランチやPRを作成せずに直接実装して完了とします。

**用途:**
- ready キューから最も優先度の高い次のタスクを選択
- PRワークフローなしで直接実装する開発セッションを開始
- ブランチ管理なしの構造化された実行ワークフローに従う

**優先度の基準:**
1. 重要度レベル（高 → 中 → 低）
2. タグ優先度（bug → security → improvement → test → performance → refactor → docs）
3. 子タスクとブロッカーが優先化される

**典型的なフロー:**
1. 最新の変更をpull
2. すべての ready タスクを取得
3. 重要度とタグ階層を使用して優先度を評価
4. 最も優先度の高いタスクを選択して in_progress に更新
5. サブエージェント（agkan-subtask-direct）に実装を委譲
6. タスクを done に更新

### 6. agkan-subtask-direct

選択されたタスクをブランチやPRを作成せずに直接実装し、完了とします。

**用途:**
- PRワークフローなしで特定の選択されたタスクを実装
- 変更を現在のブランチに直接コミット
- 実装後にタスクステータスを done に更新

**重要:**
- このスキルはユーザーが直接呼び出すことはできません
- agkan-run-direct スキルにより内部的に使用されます
- タスクが事前に選択されていることが前提です

**典型的なフロー:**
1. タスクを実装
2. 現在のブランチに変更をコミット
3. タスクステータスを「done」に更新

### 7. agkan-review

「review」ステータスのタスクを GitHub PR ステータスと照合し、自動的に done または closed に変更します。

**用途:**
- レビュー待ちタスクの PR マージ・クローズ状況を確認
- PR 結果に基づいてタスクステータスを自動更新
- 複数のレビュータスクのバッチ処理

**典型的なフロー:**
1. すべての review タスクを取得
2. 各タスクのボディから PR URL を抽出
3. GitHub CLI で PR ステータスを確認
4. PR がマージ済みなら「done」、クローズ（マージなし）なら「closed」に移行

### 8. agkan-planning-subtask

単一のバックログタスクを確認し、分解度、実装準備状態、優先順位の順序を評価します。

**用途:**
- 単一のバックログタスクの内容確認と完全性の確認
- タスクのさらなる細分化が必要かどうかの判断
- 準備完了したタスクを「ready」ステータスに移行
- 保留すべきタスクのタグ付け

**重要:**
- このスキルはユーザーが直接呼び出すことはできません
- agkan-planning スキルにより内部的に使用されます
- タスクが事前に選択されていることが前提です

**典型的なフロー:**
1. タスク内容を確認し、調査結果を補足
2. タスクの分解が必要かどうか判断（1タスク = 1PR = 1機能）
3. 実装が明確で障害物がない場合、「ready」ステータスに移行
4. 保留可能な場合、「will-do-later」タグを付与

### 9. release

バージョンアップ、CHANGELOG 更新、タグ打ち、プッシュ、オプションで npm publish までのリリースプロセス全体を自動化します。

**用途:**
- プロジェクトのバージョンリリースを実施
- バージョンバンプ、git タグ、プッシュを一連のワークフローで自動化
- リリースノートの生成と CHANGELOG の更新
- デプロイの終端として npm publish を実行

**バージョン種別:**
- **patch**: バグ修正と後方互換の改善
- **minor**: 後方互換の新機能追加
- **major**: 破壊的変更

**典型的なフロー:**
1. バージョンバンプ種別を決定（patch/minor/major）
2. テストとビルドを実行してプロジェクトの健全性を確認
3. CHANGELOG.md を更新（Unreleased セクションをバージョンセクションに移動）
4. `npm version <type> --no-git-tag-version` で `package.json` のバージョンを更新
5. `package.json` と `CHANGELOG.md` をコミット
6. 注釈付き git タグを作成（`v1.2.0`）
7. コミットとタグを origin にプッシュ
8. オプションで `npm publish` を実行

## インストール

### マーケットプレイスを追加

まず、gendosu マーケットプレイスを Claude Code に追加します:

```bash
/plugin marketplace add gendosu/gendosu-claude-plugins
```

### プラグインをインストール

agkan-skills プラグインをインストールします:

```bash
/plugin install agkan-skills@gendosu-claude-plugins
```

マーケットプレイスが既に追加されている場合:

```bash
/plugin install agkan-skills
```

## 使用方法

### 基本的なタスク管理

ready 状態のすべてのタスクを一覧表示:
```bash
/agkan list --status ready
```

新しいタスクを作成:
```bash
/agkan add "ユーザー認証を実装" "ログインページと認証フローを追加"
```

タスクステータスを更新:
```bash
/agkan update 1 status in_progress
```

### 計画セッション

バックログタスクを確認して実装に向けて準備:
```
agkan-planning スキルを使用して、バックログタスクを確認し、実装に向けて準備してください。
```

### 実行セッション

最優先度のタスクで開発セッションを開始:
```
実行セッションを開始 - 最優先度の ready タスクを選択し、実装してプルリクエストを作成してください。
```

### タスク詳細

タスクの詳細情報を取得:
```bash
/agkan get 1 --json
```

タスク階層を表示:
```bash
/agkan list --tree
```

## ステータスワークフロー

タスクは以下のステートを通過します:

| ステータス | 説明 | 次の状態 |
|----------|------|--------|
| `icebox` | 計画前の保留アイデア | `backlog` |
| `backlog` | 新規の未確認タスク | `ready` または `closed` |
| `ready` | 明確で実装準備完了 | `in_progress` |
| `in_progress` | 実装中 | `review` |
| `review` | PR作成済み、レビュー・マージ待ち | `done` |
| `done` | 完了・マージ済み | `closed`（オプション） |
| `closed` | アーカイブ・完了済み | - |

## タグ優先度

タグはタスクの分類と優先度付けに役立ちます。複数のタスクが同じ重要度レベルを持つ場合、これらのタグ優先度を使用してください:

| 優先度 | タグ | 用途 |
|--------|------|------|
| 1 | `bug` | 緊急のバグ修正 |
| 2 | `security` | セキュリティの問題 |
| 3 | `improvement` | 機能改善 |
| 4 | `test` | テストとテストカバレッジ |
| 5 | `performance` | パフォーマンス最適化 |
| 6 | `refactor` | コードリファクタリング |
| 7 | `docs` | ドキュメント |

## 設定

プロジェクトルートの `.agkan.yml` を使用して agkan データベースパスを設定:

```yaml
path: ./.agkan/data.db
```

または環境変数を設定:

```bash
export AGENT_KANBAN_DB_PATH=/custom/path/data.db
```
